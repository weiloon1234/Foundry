# Internationalization (i18n) Guide

Foundry provides locale-aware translations with per-request locale resolution, JSON-based catalogs, and a `t!()` macro for templates.

> For per-model field translations (e.g., product name in multiple languages), see the Model Extensions guide — `HasTranslations`.

---

## Quick Start

**1. Create translation files:**

```json
// locales/en/messages.json
{
    "welcome": "Welcome",
    "greeting": "Hello, {{name}}!",
    "errors": {
        "not_found": "Resource not found",
        "unauthorized": "Please log in"
    }
}
```

```json
// locales/ms/messages.json
{
    "welcome": "Selamat datang",
    "greeting": "Helo, {{name}}!",
    "errors": {
        "not_found": "Sumber tidak dijumpai",
        "unauthorized": "Sila log masuk"
    }
}
```

**2. Config:**

```toml
# config/i18n.toml
[i18n]
default_locale = "en"
fallback_locale = "en"
resource_path = "locales"
```

**3. Use in handlers:**

```rust
use foundry::prelude::*;

async fn welcome(i18n: I18n) -> impl IntoResponse {
    let message = t!(i18n, "greeting", name = "Alice");
    Json(json!({ "message": message }))
}
// Request with Accept-Language: ms
// → { "message": "Helo, Alice!" }
```

## Translation Files

### Structure

```
locales/
├── en/
│   ├── messages.json
│   ├── validation.json
│   └── emails.json
├── ms/
│   ├── messages.json
│   └── validation.json
└── zh/
    └── messages.json
```

All `.json` files within a locale directory are merged into one catalog. Nested keys are flattened with dot notation:

Locale directories and JSON files are loaded in sorted order. Duplicate flattened
keys across files fail bootstrap with both source paths instead of using an
order-dependent winner. Locale matching is ASCII case-insensitive (`PT-br`
matches canonical `pt-BR`), and case-only duplicate locale directories are rejected.

```json
{ "errors": { "not_found": "Not found" } }
```

Becomes key: `errors.not_found`

### Placeholders

Use `{{variable}}` syntax:

```json
{
    "welcome_user": "Welcome, {{name}}!",
    "item_count": "You have {{count}} items"
}
```

## The `t!()` Macro

```rust
// Simple key
t!(i18n, "welcome")

// With one parameter
t!(i18n, "greeting", name = "Alice")

// With multiple parameters
t!(i18n, "order_summary", item = "Widget", count = "3")
```

Parameter order doesn't matter. Parameters are string-based — convert numbers with `.to_string()` or use string formatting before passing.

## Extractors

### `I18n` — Translation + Locale

The primary extractor. Resolves the locale from the request and provides translation methods:

```rust
async fn handler(i18n: I18n) -> impl IntoResponse {
    let locale = i18n.locale();           // "ms"
    let text = i18n.t("welcome");          // "Selamat datang"
    let text = i18n.t_with("greeting", &[("name", "Alice")]); // "Helo, Alice!"

    Json(json!({ "locale": locale, "text": text }))
}
```

### `Locale` — Just the Locale String

When you only need the resolved locale, not the full translator:

```rust
async fn handler(Locale(locale): Locale) -> impl IntoResponse {
    Json(json!({ "current_locale": locale }))
}
```

## Locale Resolution

The locale is resolved in this order:

1. **`Locale` extension** — if set by custom middleware (e.g., from URL prefix or user preference)
2. **`Accept-Language` header** — parsed and matched against available locales
3. **Default locale** — from config (`i18n.default_locale`)

`Accept-Language` resolution honors `q` quality weights, ignores malformed or
`q=0` entries, preserves header order for equal weights, and falls back from
regional tags to base locales when possible. For example, `zh-CN` can match a
loaded `zh` catalog if there is no exact `zh-CN` catalog.

### Custom Locale Middleware

To resolve locale from a URL prefix (e.g., `/ms/dashboard`), a cookie, or user settings:

```rust
async fn locale_middleware(mut request: Request, next: Next) -> Response {
    // Example: extract from URL prefix
    let locale = request.uri().path()
        .split('/')
        .nth(1)
        .unwrap_or("en")
        .to_string();

    request.extensions_mut().insert(Locale(locale));
    next.run(request).await
}
```

## Fallback Chain

When a key is missing from the requested locale:

1. **Requested locale** — look up `"errors.not_found"` in `ms` catalog
2. **Fallback locale** — look up in `en` catalog
3. **Key itself** — return `"errors.not_found"` as-is (useful when English keys are the default)

This means you can start with English keys as your translation keys and only add translation files for non-English locales.

## Direct Manager Access

Outside of HTTP handlers (e.g., in jobs, schedules):

```rust
let i18n = app.i18n()?;

let translated = i18n.translate("ms", "welcome", &[]);
let translated = i18n.translate("en", "greeting", &[("name", "Alice")]);

// Check available locales
let locales = i18n.locale_list();  // ["en", "ms", "zh"]
let has_ms = i18n.has_locale("ms"); // true

// Parse Accept-Language header
let locale = i18n.resolve_locale("ms,en-US;q=0.9,en;q=0.8");  // "ms"
```

## Validation Messages

The validation system automatically uses i18n for error messages. Create a `validation.json` per locale:

```json
// locales/en/validation.json
{
    "required": "The {{attribute}} field is required",
    "email": "The {{attribute}} field must be a valid email",
    "min": "The {{attribute}} field must be at least {{min}} characters"
}
```

```json
// locales/ms/validation.json
{
    "required": "Ruangan {{attribute}} diperlukan",
    "email": "Ruangan {{attribute}} mesti e-mel yang sah",
    "min": "Ruangan {{attribute}} mesti sekurang-kurangnya {{min}} aksara"
}
```

The validator resolves messages based on the request locale automatically.

## Config Reference

```toml
[i18n]
default_locale = "en"         # used when locale can't be resolved
fallback_locale = "en"        # used when key missing in requested locale
resource_path = "locales"     # directory containing locale folders
```

If `resource_path` doesn't exist, i18n initializes with an empty catalog (no error). This makes i18n optional — apps that don't need translations simply skip the directory.
